Creating Agent Dialogues

Alice and Bob: KQML and KIF Communication

Activity: Formative | Type: Agent Dialogue Creation | Status: ✓ Complete

Activity Overview

Task

Create a KQML dialogue between two agents:

  • Alice: Stock procurement agent (buyer)
  • Bob: Warehouse stock control agent (supplier)

Dialogue Requirements

  • Alice asks Bob about available stock of 50 inch televisions
  • Alice queries the number of HDMI slots on the televisions
  • Use KQML performatives for communication
  • Use KIF (Knowledge Interchange Format) for knowledge representation

Learning Connection

This activity brings together concepts from my Collaborative Discussion 2 where I explored how agents communicate using structured messages with clear intentions. Now I'm creating an actual dialogue to demonstrate this understanding.

Alice and Bob Dialogue

The Scenario

Alice is a procurement agent who needs to check if the warehouse has 50 inch TVs in stock. She has two questions:

  • Are there 50 inch TVs available?
  • How many HDMI slots do they have?

So Alice reaches out to Bob, who manages the warehouse inventory, to get these answers.

How Alice and Bob Communicate

Step 1: Alice Asks Bob About Stock

Alice sends Bob a message asking: "How many 50 inch televisions do you have in stock?"

(ask
  :sender alice
  :receiver bob
  :content (quantity-in-stock ?qty Television-50inch)
  :reply-with stock-query-1
)

What's happening: Alice uses the KQML performative "ask" to request information. The KIF content says "I want to know the quantity (?qty) of 50 inch televisions in stock." The :reply-with tag helps track this conversation.

Step 2: Bob Answers with the Stock Level

Bob checks his inventory system and responds: "I have 24 of those 50 inch TVs available."

(reply
  :sender bob
  :receiver alice
  :in-reply-to stock-query-1
  :content (quantity-in-stock 24 Television-50inch)
)

What's happening: Bob uses "reply" to answer Alice's question. Instead of the variable ?qty, Bob provides the actual number: 24. The :in-reply-to tag connects this message to Alice's original ask.

Step 3: Alice Asks About the HDMI Specifications

Now Alice needs to know technical details: "By the way, how many HDMI slots do those TVs have?"

(ask
  :sender alice
  :receiver bob
  :content (hdmi-slots ?slots Television-50inch)
  :reply-with hdmi-query-1
)

What's happening: Alice asks another question using "ask" again. This time the variable ?slots represents the number of HDMI ports. It's a new query, so it has a different :reply-with tag (hdmi-query-1).

Step 4: Bob Provides the HDMI Information

Bob responds: "The 50 inch TVs have 3 HDMI slots each."

(reply
  :sender bob
  :receiver alice
  :in-reply-to hdmi-query-1
  :content (hdmi-slots 3 Television-50inch)
)

What's happening: Bob replies with the concrete answer. He replaces the variable ?slots with the actual number 3, giving Alice the information she needs.

Why This Works: Breaking It Down

Why Use KQML Here?

In my CD2 discussion, I learned that performatives are about intention. Notice what happens in this dialogue:

  • Alice uses "ask": She's not commanding Bob. She's asking politely, and Bob could refuse if he wanted to.
  • Bob uses "reply": Bob chooses to answer. He's not forced—he decides to cooperate.

This is very different from a Java method call where you just order something to happen. Here, Alice and Bob are agents with autonomy.

Why Use KIF for the Content?

KIF is a formal way to write what the agents know. Why does this matter?

  • Variables like ?qty: These represent unknowns. Alice says "I want to know ?qty" and Bob responds with the actual value "24".
  • Clear meaning: Instead of ambiguous text, KIF says exactly what we're talking about: "Television-50inch" not just "TV" or "50 inch" (which could mean different things).
  • No confusion: Both agents understand that "Television-50inch" means the same thing. This solves the ontology problem I discussed in my CD2.

Agent Autonomy in Practice

Here's what makes this realistic: Bob didn't have to answer!

  • Bob could have said: "I don't have time to answer questions right now."
  • Bob could have asked: "Why do you need this information?"
  • Bob could have negotiated: "I'll tell you the stock if you agree to minimum order quantities."

But Bob chose to cooperate. That's agent autonomy—the ability to make decisions, not just execute commands.

Real-World Example

Imagine this is actually happening:

  • Alice could be a procurement software system for a retail store
  • Bob could be an inventory management system in a warehouse
  • They communicate through KQML to keep stock levels optimal
  • If stock drops below a threshold, Alice automatically asks Bob for more information
  • Bob checks his database and replies with accurate information
  • Alice uses this info to decide whether to place an order

All of this happens without human intervention. That's the power of agent-based systems.

What This Shows About Agent Computing

Why Use Agents? (The Motivation)

This dialogue with Alice and Bob shows why agent-based systems are actually useful:

  • Modularity: Alice and Bob can be built, updated, and maintained separately. You don't need one giant program.
  • Autonomy: Each agent makes its own decisions. Alice decides when to ask questions. Bob decides what to answer.
  • Scalability: This pattern works for 2 agents (Alice and Bob) or 2,000 agents in a huge corporation. You just add more agents.
  • Real Intelligence: Unlike a simple function call, agents can negotiate, refuse, prioritize, and adapt. They're not just robots.

What Kind of Agents Are Alice and Bob?

From studying agent models, I can see both types in action:

  • Bob is Reactive: Someone asks, Bob checks his database and replies. Simple stimulus-response. It's fast and doesn't require complex reasoning.
  • Alice is Deliberative: Alice has a goal (find out TV stock) and thinks about how to achieve it. She decided to ask Bob. She could ask other agents. She's reasoning about her actions.
  • They Communicate with Standards: KQML is grounded in AI research on how humans communicate (speech act theory). Using formal standards (KQML + KIF) means systems from different companies can work together.

How My Discussion Connected to This Activity

Creating this dialogue brought together everything from my CD2:

  • I understood "ask" and "tell": Now I've used them in a real dialogue. I see how Alice uses "ask" because she respects Bob's autonomy.
  • I knew about the ontology problem: Now I've seen how KIF solves it. By saying "Television-50inch" explicitly, there's no confusion about what we mean.
  • I read about KQML performatives: Now I've written them. I understand why the :sender, :receiver, and :reply-with tags matter.
  • I studied agent autonomy: Creating this dialogue forced me to think about: What if Bob said no? What would that look like? (It would still be valid!)

The Big Picture

This activity makes clear why agent-based computing matters in the real world:

Modern systems are too complex for one program to handle. You need independent agents (Alice, Bob, and others) that can communicate reliably, maintain autonomy, and work together. KQML gives them a language. KIF makes sure they understand each other. Together, they solve real problems—like managing warehouse inventory without human intervention.

What Creating This Dialogue Taught Me

From Reading About KQML to Actually Using It

In seminar prep, I studied examples. But actually creating one is different. I had to think about:

  • How do I structure a message so Bob understands exactly what Alice wants?
  • Why does :reply-with matter? (Because with multiple questions, you need to track which reply goes with which ask.)
  • How do I use KIF variables to represent unknowns? (Alice asks about ?qty, Bob fills in 24.)

Just reading examples doesn't make this clear. Creating one forced me to understand the mechanics.

Real-World Thinking

While writing this, I started thinking about edge cases:

  • What if Bob didn't have that model of TV in stock? (His reply would be different.)
  • What if "50 inch" is ambiguous? (That's why KIF defines it precisely—Television-50inch is unambiguous.)
  • What if Bob wanted to negotiate? ("I'll tell you the stock if you commit to buying at least 10 units.")

This made me realize that real agent systems need to handle these complexities, not just the happy path.

Why KQML and KIF Matter Together

I finally understand why you need both:

  • KQML (the structure): Tells you how agents talk—with clear sender, receiver, and intention.
  • KIF (the content): Tells you what they're talking about with precision and no ambiguity.

KQML without KIF would be like having a conversation where you know the grammar but can't understand the words. KIF without KQML would be like having detailed information with no way to request it.
